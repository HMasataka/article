## 論理削除の辛さ

退会したユーザーや削除した投稿を「残しつつ消えたことにする」要件で、よく使われるのが論理削除。`deleted_at TIMESTAMP NULL` のような列を足して、`WHERE deleted_at IS NULL` でフィルタする手法。

導入は手軽だが、運用していくとじわじわ辛くなる。

- クエリ毎に `WHERE deleted_at IS NULL` を書く必要がある。一箇所でも忘れると削除済みも返ってしまう
- ORM のデフォルトスコープで隠したら、今度は削除済みを含めたい場面で抜け道を作ることになる
- ユニーク制約が機能しなくなる。同じメールアドレスが削除済みと未削除で2件並べる
- 外部キーの参照先が生きているかを DB では保証できなくなる
- 削除済みレコードが本番テーブルに溜まり、インデックスや実行計画にも影響する

元を辿ると、ライフサイクルの違う状態を 1 つのテーブルに同居させているのが原因。在籍中ユーザーと退会済みユーザーでは、扱う情報も参照頻度も違う。それを 1 つのテーブルに NULL 許容列でまとめるところに無理が出る。

この記事では、論理削除を使わずに状態をテーブル単位で分ける設計を書く。具体例はユーザーの退会。

## 状態を列ではなくテーブルで表す

「ユーザーが退会した」を表現する方法はだいたい3つに分かれる。

1. 論理削除: `users.deleted_at` に時刻を入れる
2. イベントテーブル: `user_withdrawals(user_id, withdrawn_at, reason)` に行を入れる。`users` 自体は触らない
3. 状態別テーブル: `users` から `withdrawn_users` にレコードを移す

(2) と (3) はどちらも状態をテーブルで表現する派だが、思想が違う。

- (2) イベント志向: 「いつ・なぜ退会したか」を履歴として残したい。状態の戻りが頻繁にあるケースに向く
- (3) 状態志向: 在籍中と退会済みを別物として扱いたい。状態は「今どのテーブルにいるか」で決まる

実際は両方併用することも多い。この記事は (3) を中心に書く。

## ひもづくレコードまで一緒に移さない

(3) を採用しようとすると最初に踏みがちなのが、「ユーザーを `withdrawn_users` に移すなら、その人の投稿 1000 件もぜんぶ `withdrawn_posts` に移すべきでは?」という発想。

これはやらない方が良い。投稿の状態と投稿者の状態は別物として管理する方が素直になる。

投稿のライフサイクルは「下書き → 公開 → 非公開 → 削除」のように投稿側の都合で動く。投稿者が退会したかどうかと、その投稿が削除されるべきかは独立した話。連鎖させようとすると、

- 投稿に紐づくコメント、いいね、タグも追従が必要になり、雪だるま式にテーブルが二重化する
- 退会ユーザーの投稿に他ユーザーがいいねしていた場合、その「いいね」はどっち側に置くべきか決められない
- 「全投稿一覧」を取るたびに `posts UNION posts_archived` のクエリになり、インデックスも二重化する

テーブル分割で表したいのはそのエンティティ自身のライフサイクルだけ。ひもづくデータの状態はそれぞれのライフサイクルで独立して管理する。

「退会ユーザーの投稿を見せたくない」という要件はデータを移すのではなく、クエリ側で在籍中ユーザーに絞ることで対応する。後述。

## MySQL での実装

素直に書くとこうなる。

```sql
CREATE TABLE user_accounts (
  id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
);

CREATE TABLE users (
  user_account_id BIGINT UNSIGNED PRIMARY KEY,
  email           VARCHAR(255) NOT NULL UNIQUE,
  display_name    VARCHAR(255) NOT NULL,
  FOREIGN KEY (user_account_id) REFERENCES user_accounts(id)
);

CREATE TABLE withdrawn_users (
  user_account_id BIGINT UNSIGNED PRIMARY KEY,
  withdrawn_at    DATETIME NOT NULL,
  reason          VARCHAR(255),
  FOREIGN KEY (user_account_id) REFERENCES user_accounts(id)
);

CREATE TABLE posts (
  id              BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  user_account_id BIGINT UNSIGNED NOT NULL,
  body            TEXT NOT NULL,
  FOREIGN KEY (user_account_id) REFERENCES user_accounts(id)
);
```

ポイントは `user_account_id` を発番する親テーブル `user_accounts` を別に置くこと。投稿などひもづくテーブルはこの親を参照する。在籍中か退会済みかに関わらずユーザーの一意 ID は親が持つので、参照先が状態によって切り替わる、というややこしさが消える。

`users` と `withdrawn_users` はそれぞれの状態に固有のカラムだけを持つ。`users` には `email` や `display_name`、`withdrawn_users` には `withdrawn_at` や `reason`。NULL 許容列で同居させなくて済む。

### 同じユーザーが両方に入らないことを保証する

「同じ `user_account_id` が `users` と `withdrawn_users` の両方に同時に存在しないことをどう保証するか」という問題が出てくる。

MySQL では、状態を判別子に組み込んだ複合外部キーを使うとトリガーなしで保証出来る。

```sql
CREATE TABLE user_accounts (
  id    BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  state ENUM('active','withdrawn') NOT NULL,
  UNIQUE KEY uk_id_state (id, state)
);

CREATE TABLE users (
  user_account_id BIGINT UNSIGNED PRIMARY KEY,
  state           ENUM('active') GENERATED ALWAYS AS ('active') STORED,
  email           VARCHAR(255) NOT NULL UNIQUE,
  FOREIGN KEY (user_account_id, state)
    REFERENCES user_accounts(id, state)
);

CREATE TABLE withdrawn_users (
  user_account_id BIGINT UNSIGNED PRIMARY KEY,
  state           ENUM('withdrawn') GENERATED ALWAYS AS ('withdrawn') STORED,
  withdrawn_at    DATETIME NOT NULL,
  FOREIGN KEY (user_account_id, state)
    REFERENCES user_accounts(id, state)
);
```

仕組みは下記の通り。

- `user_accounts.state` は `active` か `withdrawn` のどちらか一方しか持てない
- `users.state` は生成列で常に `'active'` 固定。複合 FK で `(id, 'active')` を要求する
- `withdrawn_users.state` も同様に `(id, 'withdrawn')` を要求する
- 親が `'active'` のときに `withdrawn_users` に INSERT すると FK 違反でエラーになる

これでトリガーもアプリ側のチェックも無しに「両方には入らない」が成立する。

なお「どちらかには必ず存在する」側の保証は MySQL の制約だけでは難しい(deferred constraint が無いため)。これは `user_accounts` と `users` を必ず同一トランザクションで INSERT する、というアプリ規約で担保する。

### 退会への遷移

`users` から `withdrawn_users` への移動はトランザクションで包む。順序に注意。

```sql
START TRANSACTION;
  DELETE FROM users WHERE user_account_id = ?;
  UPDATE user_accounts SET state = 'withdrawn' WHERE id = ?;
  INSERT INTO withdrawn_users (user_account_id, withdrawn_at, reason)
    VALUES (?, NOW(), ?);
COMMIT;
```

親の `state` を先に更新すると `users` の FK が外れて失敗するので、子テーブルから先に削除する。トランザクションの分離のおかげで、外から「両方に存在しない瞬間」を観測されることはない。

## クエリは INNER JOIN がそのままフィルタになる

「フォロー中ユーザーの投稿一覧から退会済みユーザー分を除く」というよくあるクエリを考える。論理削除の場合、

```sql
SELECT p.*
FROM   posts p
JOIN   follows f ON f.followee_id = p.user_id
JOIN   users   u ON u.id = p.user_id
WHERE  f.follower_id = ?
  AND  u.deleted_at IS NULL;          -- 書き忘れたら退会者も出る
```

となる。`AND u.deleted_at IS NULL` を全クエリで書き続ける必要があり、一箇所でも忘れたらバグ。

状態別テーブルの設計だと下記。

```sql
SELECT p.*
FROM   posts p
JOIN   follows f ON f.followee_id = p.user_account_id
JOIN   users   u ON u.user_account_id = p.user_account_id   -- INNER JOIN が状態フィルタ
WHERE  f.follower_id = ?;
```

`WHERE u.state = 'active'` がどこにも出てこない。「在籍中ユーザーの投稿」という意図が `users` への INNER JOIN そのものに表れていて、書き忘れる余地が無い。

逆に「退会済みユーザーの投稿を管理画面に出す」なら下記のように JOIN 先を差し替えるだけで済む。

```sql
SELECT p.*, w.withdrawn_at
FROM   posts p
JOIN   withdrawn_users w ON w.user_account_id = p.user_account_id;
```

`suspended_users`(凍結中)のような状態を増やしても、既存クエリには影響しない。これは論理削除では得られない性質。

### `user_accounts.state` を持たせるかどうか

前述の複合 FK を使うと `user_accounts.state` という列を導入することになる。「結局カラムで状態を持っているのと同じでは?」という疑問は出る。

半分はその通り。設計判断としては 2 択になる。

- `state` 列あり: DB だけで排他制約を保証出来る。ただし状態判別を列参照でやる場面が出てくる
- `state` 列なし: 排他制約はトランザクション規約に任せる。状態は「どのテーブルに行があるか」だけで決まり、クエリの書き心地は良い

DB レベルで不変条件を担保したいなら前者、クエリの読みやすさを優先するなら後者。チームの文化次第。

## イベントテーブルで履歴を残す

退会と復帰を繰り返すケースや、「いつ・なぜ」の履歴自体に業務的な意味があるケースでは、状態遷移をイベントとして追記する方式が向く。append-only な `user_events` を作って、退会・復帰のたびにレコードを足していく。

```sql
CREATE TABLE user_events (
  id          BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  user_id     BIGINT UNSIGNED NOT NULL,
  event_type  ENUM('withdrew','reactivated') NOT NULL,
  occurred_at DATETIME NOT NULL,
  reason      VARCHAR(255),
  FOREIGN KEY (user_id) REFERENCES users(id),
  KEY idx_user_occurred (user_id, occurred_at)
);
```

現在の状態は「そのユーザーの最新イベント」で決まる。アクティブは「最新イベントが `reactivated`、または1度もイベントが無い」と定義出来る。

```sql
SELECT u.*
FROM   users u
LEFT JOIN (
  SELECT e.user_id, e.event_type
  FROM   user_events e
  JOIN  (
    SELECT user_id, MAX(occurred_at) AS latest
    FROM   user_events
    GROUP  BY user_id
  ) lat ON lat.user_id = e.user_id AND lat.latest = e.occurred_at
) cur ON cur.user_id = u.id
WHERE  cur.event_type IS NULL OR cur.event_type = 'reactivated';
```

問題はクエリ毎にこれを書くこと。素朴にやるとユーザー一覧を引くたびに最新イベントの集約が走り、当然遅い。

回避策は大きく 2 つ。

- `users.current_state` のような列を持って、イベント挿入時にトランザクションで一緒に更新する。クエリは `WHERE current_state = 'active'` で済むが、論理削除と似た形になり書き忘れのリスクが戻る
- 状態別テーブル(`users` / `withdrawn_users`)と併用する。状態の「今」はテーブル分割で表現し、遷移の履歴は `user_events` に残す

後者の併用なら、書き忘れ防止のメリットと、履歴を残せる監査側のメリットを両取り出来る。代わりにテーブル間の整合性はアプリ側のトランザクション規約頼みになる。

```sql
START TRANSACTION;
  DELETE FROM users WHERE user_account_id = ?;
  UPDATE user_accounts SET state = 'withdrawn' WHERE id = ?;
  INSERT INTO withdrawn_users (user_account_id, withdrawn_at, reason)
    VALUES (?, NOW(), ?);
  INSERT INTO user_events (user_id, event_type, occurred_at, reason)
    VALUES (?, 'withdrew', NOW(), ?);
COMMIT;
```

復帰は逆方向の遷移に `'reactivated'` イベントを足すだけ。

## トレードオフと使いどき

良い点は、クエリから状態フィルタが消えること、状態ごとに必要なカラムだけを持てること、ユニーク制約が素直に効くこと、状態が増えても既存クエリが壊れないこと。

辛い点は、遷移処理が `DELETE` + `UPDATE` + `INSERT` の複数文になること、全状態を横断するクエリで `UNION ALL` が必要になること、ORM の自動マッピングと相性が悪いこと、既存システムへの後追い導入は移行コストが大きいこと。

向くのは、状態でカラム構成が大きく違う、状態遷移が一方通行に近い、状態の混入が業務バグになる(退会者をメール配信対象に入れる等)、といったケース。逆に状態遷移が頻繁に行き来する(下書き ⇄ 公開 等)、状態ごとのカラム差が無い、全状態横断クエリが主要、といったケースでは論理削除や状態列で十分。

履歴自体に業務的な意味がある(「いつ・誰が・なぜ」を残したい、監査要件がある)場合はイベントテーブル方式が向く。実際は「本体は状態別テーブル、遷移履歴はイベントテーブルにも残す」というハイブリッドが落とし所になることも多い。

## まとめ

論理削除は手軽だが、ライフサイクルの違うものを 1 つのテーブルに同居させているという根本問題から、運用コストが時間と共に増えていく。状態別テーブルで設計すると下記が成立する。

- ひもづくレコードを連鎖移動させない(関心の分離)
- 排他制約は親テーブル + 複合 FK か、トランザクション規約で守る
- クエリは INNER JOIN がそのまま状態フィルタになる

書き忘れによる事故が構造上消え、状態が増えても既存クエリが壊れない。

`deleted_at` で消す前に、「これは本当に削除なのか、別の状態への遷移なのか」を一度考えてみると良い。多くの場合、それは削除ではなく別のライフサイクル段階への移動になっている。
