## はじめに

業務システムを作っていると、退会したユーザーや削除した記事を「消えたことにしたいけれど履歴は残したい」という要件によく出会うかと思います。
このときによく見るのが論理削除と呼ばれる手法で、`deleted_at` のような列を足して `WHERE deleted_at IS NULL` で取り出すというものです。
導入は手軽な反面、実際に運用していくとじわじわと辛さが出てくる手法でもあるかなと思っています。

クエリの度に `WHERE deleted_at IS NULL` を書く必要があり、一箇所でも書き忘れると削除済みのレコードが返ってきてしまいます。
ORM のデフォルトスコープで隠すという手もありますが、今度は削除済みを含めたい場面で抜け道を作ることになります。
さらに、同じメールアドレスが削除済みと未削除で 2 件並べてしまえるためユニーク制約が機能しなくなる、外部キーの参照先が生きているかを DB では保証できなくなる、削除済みレコードが本番テーブルに溜まり続けてインデックスや実行計画に影響してくる、といった問題も次第に効いてきます。

元を辿ると、ライフサイクルの異なる状態を 1 つのテーブルに同居させているところが原因ではないかなと思います。
在籍中のユーザーと退会済みのユーザーでは、扱う情報も参照頻度も違うので、それを 1 つのテーブルに NULL 許容列でまとめるところに無理が出ているのではないでしょうか。

本記事では、論理削除に頼らずに状態をテーブル構造そのもので表現する設計について、ユーザーの退会を題材に書いていこうと思います。

## 状態を表現する 3 つの選択肢

「ユーザーが退会した」という事実をテーブルで表現する方法は、ざっくり 3 つに分かれるかと思います。

1. 論理削除を使い、`users.deleted_at` に時刻を入れる
2. イベントテーブルを用意し、`user_events` に「退会した」「復帰した」という行を追加していく
3. 状態別テーブルを用意し、`users` から `withdrawn_users` にレコードを移す

(1) の論理削除については先ほど触れた通りなので、本記事では (2) と (3) について書いていきます。
どちらも状態をテーブルで表現する派ですが、思想は少し違っていて、(2) は「いつ・なぜその状態になったのか」を履歴として残したい場合に向いている方式です。退会と復帰を繰り返すようなケースとも相性が良いと思います。
一方の (3) は、在籍中ユーザーと退会済みユーザーを別物として扱いたい場合に向いていて、状態は「今どのテーブルに行があるか」で決まる形になります。

実務では両方を組み合わせる場面も出てくるのですが、まずは (3) の状態別テーブルから順に見ていこうと思います。

## 状態別テーブルで設計する

最初に気をつけておきたいのが、ユーザーを退会用のテーブルに移すからといって、そのユーザーが書いた投稿やコメントまで連鎖的に動かさないことです。
僕も以前関わったプロジェクトで踏みかけたことがあるのですが、「ユーザーを `withdrawn_users` に移すなら、その人の投稿 1000 件も全部 `withdrawn_posts` に移すべきでは?」と考えてしまいがちです。
これをやってしまうと、投稿に紐づくコメントもいいねもタグも全て追従させる必要が出てきて、雪だるま式にテーブルが二重化していきます。
退会ユーザーの投稿に他のユーザーがいいねしていた場合、そのいいねを退会側に置くのか在籍側に置くのかも決められなくなります。

テーブル分割で表現したいのは、そのエンティティ自身のライフサイクルだけだと思います。
投稿のライフサイクルは「下書き → 公開 → 非公開 → 削除」のように投稿側の都合で動くので、投稿者が退会したかどうかとは独立して扱うのが素直だと思います。
「退会ユーザーの投稿を見せたくない」という要件は、データを移すのではなくクエリ側で在籍中ユーザーに絞ることで解決するのですが、これは後ほど書きます。

具体的なスキーマを見ていきます。
ユーザーを在籍中と退会済みの 2 テーブルに分けるとして、まず大事になるのが ID を発番する親テーブルを別に置くことです。

```sql
CREATE TABLE user_accounts (
  id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
);

CREATE TABLE users (
  user_account_id BIGINT UNSIGNED PRIMARY KEY,
  email           VARCHAR(255) NOT NULL UNIQUE,
  display_name    VARCHAR(255) NOT NULL,
  FOREIGN KEY (user_account_id) REFERENCES user_accounts(id)
);

CREATE TABLE withdrawn_users (
  user_account_id BIGINT UNSIGNED PRIMARY KEY,
  withdrawn_at    DATETIME NOT NULL,
  reason          VARCHAR(255),
  FOREIGN KEY (user_account_id) REFERENCES user_accounts(id)
);

CREATE TABLE posts (
  id              BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  user_account_id BIGINT UNSIGNED NOT NULL,
  body            TEXT NOT NULL,
  FOREIGN KEY (user_account_id) REFERENCES user_accounts(id)
);
```

`user_account_id` を発番するのが `user_accounts` で、投稿などひもづくテーブルはこの親を参照します。
こうしておくと、在籍中か退会済みかに関わらずユーザーの一意 ID は親が持つことになるので、参照先テーブルが状態によって切り替わるといったややこしさが消えます。
`users` と `withdrawn_users` はそれぞれの状態に固有のカラムだけを持つので、`withdrawn_at` を在籍中のユーザーに NULL で持たせる必要もありません。

### 両方のテーブルに同時に入らないことを保証する

次に気になるのが、「同じ `user_account_id` が `users` と `withdrawn_users` の両方に同時に存在しないことをどう保証するか」だと思います。
MySQL には deferred constraint が無いので一気に解決するのは難しいのですが、状態を判別子として外部キーに組み込んでしまうと、トリガーなしで保証できるようになります。

```sql
CREATE TABLE user_accounts (
  id    BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  state ENUM('active','withdrawn') NOT NULL,
  UNIQUE KEY uk_id_state (id, state)
);

CREATE TABLE users (
  user_account_id BIGINT UNSIGNED PRIMARY KEY,
  state           ENUM('active') GENERATED ALWAYS AS ('active') STORED,
  email           VARCHAR(255) NOT NULL UNIQUE,
  FOREIGN KEY (user_account_id, state)
    REFERENCES user_accounts(id, state)
);

CREATE TABLE withdrawn_users (
  user_account_id BIGINT UNSIGNED PRIMARY KEY,
  state           ENUM('withdrawn') GENERATED ALWAYS AS ('withdrawn') STORED,
  withdrawn_at    DATETIME NOT NULL,
  FOREIGN KEY (user_account_id, state)
    REFERENCES user_accounts(id, state)
);
```

`user_accounts.state` はある時点で `active` か `withdrawn` のどちらか一方しか持てません。
`users.state` は生成列で常に `'active'` 固定になっており、複合 FK で `(id, 'active')` を要求します。`withdrawn_users.state` も同じく `(id, 'withdrawn')` を要求します。
親の `state` が `'active'` のときに `withdrawn_users` に INSERT しようとすると FK 違反でエラーになるので、両方のテーブルに同時に行が入ることが構造的にあり得なくなります。

ここまで来ると `user_accounts.state` を見れば現在の状態が分かるので、「結局カラムで状態を持っているのと同じでは?」という疑問が出てくるかもしれません。
正直なところ半分はその通りで、ここはチームの好みで 2 択になるかと思います。
`state` 列を持たせれば DB だけで排他制約を担保できますが、状態判別が列参照になる場面が出てきます。
持たせない場合は排他制約をトランザクション規約に任せる必要が出てきますが、状態は「どのテーブルに行があるか」だけで決まる素直な構造になり、クエリの書き心地は良くなります。
DB レベルで不変条件を担保したいなら前者、クエリの読みやすさを優先するなら後者、といった感じだと思います。

### 退会への遷移

退会の処理はトランザクションで包みます。順序に気をつけて、子テーブルから削除する必要があります。

```sql
START TRANSACTION;
  DELETE FROM users WHERE user_account_id = ?;
  UPDATE user_accounts SET state = 'withdrawn' WHERE id = ?;
  INSERT INTO withdrawn_users (user_account_id, withdrawn_at, reason)
    VALUES (?, NOW(), ?);
COMMIT;
```

親の `state` を先に更新してしまうと `users` の FK が外れて UPDATE で失敗するので、`users` から削除する → 親の `state` を更新する → `withdrawn_users` に INSERT する、の順番で並べています。
トランザクションの分離のおかげで、外から「両方のテーブルに行が無い瞬間」が観測されることはありません。

## クエリは INNER JOIN がそのままフィルタになる

状態別テーブルにする良さが特に効いてくるのがクエリの書き方かなと思います。
例えば「フォロー中ユーザーの投稿一覧から、退会済みユーザーの分は除く」というクエリを考えます。
論理削除で同じことをやろうとすると下記のような形になります。

```sql
SELECT p.*
FROM   posts p
JOIN   follows f ON f.followee_id = p.user_id
JOIN   users   u ON u.id = p.user_id
WHERE  f.follower_id = ?
  AND  u.deleted_at IS NULL;          -- 書き忘れたら退会者も出てしまう
```

`AND u.deleted_at IS NULL` が必須で、一箇所でも書き忘れたらバグになってしまいます。
状態別テーブルで設計しておくと、ここは下記のようになります。

```sql
SELECT p.*
FROM   posts p
JOIN   follows f ON f.followee_id = p.user_account_id
JOIN   users   u ON u.user_account_id = p.user_account_id   -- INNER JOIN そのものがフィルタ
WHERE  f.follower_id = ?;
```

`WHERE u.state = 'active'` がどこにも出てきません。
「在籍中ユーザーの投稿」という意図が `users` テーブルへの INNER JOIN そのものに表れていて、書き忘れる余地がそもそも存在しないというのが嬉しいところです。
逆に「退会済みユーザーの投稿を管理画面に出す」という要件であれば、JOIN 先を `withdrawn_users` に差し替えるだけで済みます。

```sql
SELECT p.*, w.withdrawn_at
FROM   posts p
JOIN   withdrawn_users w ON w.user_account_id = p.user_account_id;
```

将来 `suspended_users` (凍結中) のような状態を増やすことになっても、既存のクエリは一切壊れません。
状態フィルタが構文として表現されているのが、この設計でいちばん効いてくる部分かなと思います。

## イベントテーブルで履歴を残す

ここまでは選択肢 (3) の状態別テーブルの話でしたが、システムによっては「いつ・なぜ退会したのか」「いつ復帰したのか」という履歴自体に業務的な意味があるケースもあるかと思います。
監査要件があるケースや、ユーザーが退会と復帰を繰り返すような SaaS だと、選択肢 (2) のイベントテーブル方式が素直に書けると思います。

`user_events` のような append-only のテーブルを用意して、退会・復帰のたびにレコードを足していくだけです。

```sql
CREATE TABLE user_events (
  id          BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  user_id     BIGINT UNSIGNED NOT NULL,
  event_type  ENUM('withdrew','reactivated') NOT NULL,
  occurred_at DATETIME NOT NULL,
  reason      VARCHAR(255),
  FOREIGN KEY (user_id) REFERENCES users(id),
  KEY idx_user_occurred (user_id, occurred_at)
);
```

現在の状態はそのユーザーの最新イベントで決まる形になり、アクティブなユーザーは「最新イベントが `reactivated`、または 1 度もイベントが無い」と定義できます。
ただ、これをクエリで取り出そうとすると下記のように最新イベントの集約が必要になります。

```sql
SELECT u.*
FROM   users u
LEFT JOIN (
  SELECT e.user_id, e.event_type
  FROM   user_events e
  JOIN  (
    SELECT user_id, MAX(occurred_at) AS latest
    FROM   user_events
    GROUP  BY user_id
  ) lat ON lat.user_id = e.user_id AND lat.latest = e.occurred_at
) cur ON cur.user_id = u.id
WHERE  cur.event_type IS NULL OR cur.event_type = 'reactivated';
```

これをユーザー一覧を引くたびに走らせるのはかなり辛く、パフォーマンスにも当然影響してきます。
回避策は大きく 2 つあるかなと思います。

ひとつ目は、`users.current_state` のような列を持たせて、イベント挿入時にトランザクションで一緒に更新する方法です。
クエリ側は `WHERE current_state = 'active'` で済むようになりますが、形としては論理削除に近く、書き忘れのリスクが戻ってきてしまうのが悩ましいところです。

ふたつ目は、状態別テーブルとの併用です。
状態の「今」は `users` / `withdrawn_users` のテーブル分割で表現し、遷移の履歴は `user_events` に残す形になります。
状態判定はテーブル分割で構造化されているので書き忘れが起きず、履歴は別テーブルで読み取り専用に残せるため、個人的には履歴を残したい要件があるなら併用の構成を取ることが多いです。
代わりにテーブル間の整合性はアプリ側のトランザクション規約に乗ることになるので、書き込みを行う箇所はリポジトリ層などに統一しておく必要があります。

```sql
START TRANSACTION;
  DELETE FROM users WHERE user_account_id = ?;
  UPDATE user_accounts SET state = 'withdrawn' WHERE id = ?;
  INSERT INTO withdrawn_users (user_account_id, withdrawn_at, reason)
    VALUES (?, NOW(), ?);
  INSERT INTO user_events (user_id, event_type, occurred_at, reason)
    VALUES (?, 'withdrew', NOW(), ?);
COMMIT;
```

復帰のときも、逆方向の遷移に `'reactivated'` のイベントを足すだけです。

## 使いどころとトレードオフ

状態別テーブルが向いているのは、状態によってカラム構成が大きく異なるエンティティだと思います。
退会者には `withdrawn_at` が必須、在籍者には `last_login_at` が必須、といったケースですね。
状態遷移が一方通行に近かったり、状態の混入が業務的なバグになるケース(退会者をメール配信対象に入れてしまう、など)とも相性が良いです。
逆に状態遷移が頻繁に行ったり来たりするケース(下書き ⇄ 公開 など)や、状態ごとにカラムの差がほぼ無いケースだと、テーブルを分けるメリットは薄くなるため、論理削除や状態列の方が素直だと思います。

辛い点もいくつかあります。
退会のような遷移処理が `DELETE` + `UPDATE` + `INSERT` の複数文になるので、トランザクションを意識して書く必要が出てきます。
管理画面のように全状態を横断したい場面では `UNION ALL` が必要になりますし、状態が 4 つも 5 つもあるエンティティだとテーブルが増えすぎて取り回しが悪くなります。
ORM の自動マッピングと相性が悪いのも地味に効いてくる点で、状態ごとにモデルが分かれる構造になるためヘルパーで吸収するなどの工夫が要るかと思います。
そして、既存システムへの後追い導入は移行コストが大きいので、新規設計では推せるものの、論理削除でガッツリ書かれているシステムを途中から作り変えるのは大抵割に合わないかなと思います。

履歴自体に業務的な意味があるならイベントテーブル方式が向いていますが、現在状態を引くコストの問題があるので、実際は「本体は状態別テーブル、遷移履歴はイベントテーブルにも残す」というハイブリッドが落とし所になることが多いかと思います。

## おわりに

論理削除は手軽ですが、ライフサイクルの違うものを 1 つのテーブルに同居させているという根本的な問題から、運用コストが時間とともに膨らんでいく手法でもあるかなと思います。
状態別テーブルで設計しておくと、関連レコードまで連鎖移動させない関心の分離が自然と入りますし、排他制約は親テーブル + 複合 FK か、トランザクション規約で守れます。
クエリは INNER JOIN がそのまま状態フィルタになるので、書き忘れによる事故が構造上消えて、状態が増えても既存クエリが壊れにくくなります。
履歴を残したい要件があれば、イベントテーブルを併用すれば監査側の要求にも応えられます。

`deleted_at` で消す処理を反射的に書いてしまう前に、「これは本当に削除なのか、それとも別の状態への遷移なのか」を一度考えてみるのが良いのではないかなと思います。
多くの場合、それは削除ではなく、別のライフサイクル段階への移動になっているのではないでしょうか。
