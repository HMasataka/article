## はじめに

Goでクリーンアーキテクチャやレイヤードアーキテクチャを採用すると、usecase 層からどうやってトランザクションを発行するかという問題に必ずぶつかります。
usecaseは「複数のrepositoryをまたいだ操作を1つのトランザクションにまとめたい」というビジネス上の都合を持っている一方で、`*sql.Tx` や `pgx.Tx` のようなDB実装に直接依存させたくはありません。
この問題に対する定番の答えとして、`Transactor` のような存在を用意して `Do(func() error)` のようなメソッドでトランザクション境界をusecase側に渡す手法があります。

## Transactorパターンの基本形

まずは一番素直な形を見ていきます。
usecase層がトランザクション境界を引きたい、けれどDB実装の詳細は知りたくない、という要求に対しては次のようなインターフェースが定番です。

```go
type Transactor interface {
    Do(ctx context.Context, fn func(ctx context.Context) error) error
}
```

usecase側はこのインターフェースだけに依存します。

```go
func (u *OrderUsecase) Place(ctx context.Context, in Input) error {
    return u.tx.Do(ctx, func(ctx context.Context) error {
        if err := u.orderRepo.Save(ctx, order); err != nil {
            return err
        }
        return u.stockRepo.Decrement(ctx, item)
    })
}
```

この形にしておくと嬉しいことが3つあります。

1つめは、usecaseが `*sql.Tx` のようなDB実装に直接依存しないことです。`Transactor` の実装をdatabase/sqlで作るのか、pgxで作るのか、はたまたインメモリのno-opで作るのかはusecaseには見えません。
2つめは、テスト時に `Transactor` をモックや素通し実装に差し替えやすいことです。usecaseの単体テストで本物のトランザクションを張る必要はありません。
3つめは、トランザクション境界がusecase層に明示的に残ることです。repository側に境界を持たせると「どこからどこまでが1つのビジネス操作なのか」が曖昧になりがちですが、このパターンならビジネス境界がそのままトランザクション境界になります。

## 各repositoryが `*sql.DB` を持つ素朴な実装

`Transactor.Do` の中ではcontextに `*sql.Tx` を仕込んでおき、repositoryは「contextからtxが取れればそれを使い、なければ通常の `*sql.DB` を使う」という分岐を持つ、というのが一番素直な実装です。

```go
type orderRepo struct {
    db *sql.DB
}

func (r *orderRepo) Save(ctx context.Context, o Order) error {
    db := txFromContext(ctx, r.db) // tx があればそれ、なければ r.db
    _, err := db.ExecContext(ctx, "INSERT ...", ...)
    return err
}
```

シンプルでわかりやすく、小さなプロジェクトであればこれで十分機能します。
しかしrepositoryの数が増えてくると、いくつか気になる点が出てきます。

まず、`txFromContext` を呼んでフォールバックする、というルールがすべてのrepositoryに散らばります。新しいrepositoryを書くたびに「ここでもこの分岐を書く必要がある」と意識し続けなければなりませんし、書き忘れればトランザクションが効かないrepositoryが混ざり込みます。
また、すべてのrepositoryが `*sql.DB` を依存として保持することになるため、コンストラクタの引数や依存関係も少しずつ増えていきます。

## TransactorとDBProviderに分離する

そこで、フォールバックロジックをrepository側ではなく別の場所に寄せて、repositoryからは `*sql.DB` への直接依存を取り除く設計を考えます。

ここで欲しくなるのは「contextからtxを取り出し、なければ通常のDBコネクションを返す」というexecutor供給用のメソッドです。
ただ、これをそのまま `Transactor` に生やしてしまうと、repositoryが本来必要としていないトランザクション境界用のメソッド（`Do`）まで一緒に見えてしまいます。
usecase層が必要としているのはトランザクション境界、repository層が必要としているのはexecutor供給、と責務がきれいに分かれているので、最初からそれぞれを別のインターフェースに分けて扱うのが素直です。

```go
type DBTX interface {
    ExecContext(ctx context.Context, q string, args ...any) (sql.Result, error)
    QueryContext(ctx context.Context, q string, args ...any) (*sql.Rows, error)
    QueryRowContext(ctx context.Context, q string, args ...any) *sql.Row
}

// repository が依存するのはこっち
type DBProvider interface {
    Executor(ctx context.Context) DBTX // tx があればそれ、なければ db
}

// usecase が依存するのはこっち
type Transactor interface {
    Do(ctx context.Context, fn func(ctx context.Context) error) error
}

// 実装は同じ構造体
type txManager struct{ db *sql.DB }

func (m *txManager) Executor(ctx context.Context) DBTX        { /* ... */ }
func (m *txManager) Do(ctx context.Context, fn func(context.Context) error) error { /* ... */ }
```

repositoryは `DBProvider` だけを依存に取り、クエリを発行するときはその都度 `Executor(ctx)` を呼ぶ形になります。

```go
type orderRepo struct {
    db DBProvider // *sql.DB を直接持たない
}

func (r *orderRepo) Save(ctx context.Context, o Order) error {
    _, err := r.db.Executor(ctx).ExecContext(ctx, "INSERT ...", ...)
    return err
}
```

こうしておくと、usecaseには `Transactor` だけを、repositoryには `DBProvider` だけを渡せばよくなります。
レイヤーごとに必要な責務だけを依存として宣言できるため、インターフェース分離の原則（ISP）にも沿った形になります。
実装の本体は同じ `txManager` を使い回せるので、DIコンテナや初期化処理で2つに分ける手間も実質ありません。

フォールバックロジックも `txManager` の実装1箇所にまとまり、「contextに何が入っているか」というルールも `txManager` の内部に閉じます。
sqlcを使っている場合は `DBTX` 相当のインターフェースが自動生成されるので、この設計と相性がとても良いです。

usecase層のテストでは `Transactor` だけをモックすればよく、`Do` の中身を素通しさせる実装にしておけばトランザクションを意識せずにテストを書けます。
repository層のテストでは `DBProvider` の本物の実装に対して実DB（あるいはtestcontainers）を使ってテストを書きます。
フォールバックの分岐ロジック自体は `txManager` だけのユニットテストでカバーできるので、テストの責務分担もきれいに分かれます。

## おわりに

本記事では、Goのusecase層からトランザクションを発行する際の設計について整理してきました。要点をまとめます。

Transactorパターンの基本形として、`Do(ctx, fn)` でトランザクション境界をusecase層に渡し、DB実装の詳細をドメイン層から切り離します。
そのうえで、各repositoryが `*sql.DB` を持って `txFromContext` で分岐するのではなく、usecaseは `Transactor`、repositoryは `DBProvider` だけに依存させ、フォールバックロジックを1箇所に寄せつつレイヤーごとに責務を分けます。実装の本体は同じ構造体で構いません。

トランザクションの設計はusecase層とrepository層の責務分担に直結する重要なテーマだと感じています。
最初は素朴な実装で十分でも、repositoryの数が増えて来たり、テストの書きやすさが気になりはじめたタイミングで、今回紹介した分離パターンに寄せていくと管理しやすくなると思います。
小さく始めて、必要になったときに段階的に整理していくのが良いのではないかなと考えています。
